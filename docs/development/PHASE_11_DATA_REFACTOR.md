# Phase 11: 数据底座重构与真实数据注入

> 本文档记录了项目从“单表虚拟数据”向“多表关联真实数据”转型的完整过程。这是项目具备企业级 BI 分析能力的基石，并为未来实现“半自动化接入任意数据库”铺平了道路。

## 1. 重构背景与痛点

在旧架构中，系统仅依赖一张单表 `Sales`，且使用随机生成的测试数据。这导致了两个致命问题：
1. **分析能力受限**：无法支持真实业务中常见的跨表关联（如查“某行业客户的各类别销量”）。
2. **底层架构僵化**：Prisma 在编译时绑死了表结构，无法在运行时动态适应外部数据库，阻碍了 SaaS 化进程。
3. **体验失真**：随机生成的 "Product A" 等数据毫无业务质感，AI 生成的分析报告像无意义的废话。

## 2. 核心改造点

### 2.1 引入 Kysely，剥离 Prisma 强绑定
为了支持未来运行时的动态表名和字段映射，底层查询引擎从 Prisma 切换为 **Kysely**。
- 废弃了 `PrismaService`，新建基于 `Kysely` 和 `pg` 驱动的 `DatabaseService`。
- 保留了完整的 TypeScript 类型安全，同时获得了运行时动态拼接 SQL 的绝对灵活性。

### 2.2 建立标准 ERP 业务模型（多表关联）
将单表拆分为 4 张标准关系型表，真实反映企业业务流转：
- `Customer`：客户表（含行业、地区细分）。
- `Product`：产品表（含品类细分）。
- `SalesOrder`：订单主表（含物流、时间维度）。
- `SalesOrderItem`：订单明细表（含真实销量、金额、利润）。

### 2.3 真实业务数据注入 (ETL)
编写流式数据清洗脚本 `seed.ts`，引入经典的 Superstore 真实零售数据集：
- 使用 `csv-parser` 流式读取数万行 CSV 数据，避免内存溢出。
- 在内存中进行数据去重与外键关联，通过事务批量写入数据库。
- 最终注入约 10,000 条真实订单明细，时间跨度达 4 年，包含双十一等真实业务峰值。

### 2.4 Agent 工具能力降维打击
基于 Kysely 重写 `query_sales` 和 `gen_chart` 工具，解锁复杂 BI 能力：
- **多表 JOIN**：支持按 `Customer.region` 和 `Product.category` 进行跨表聚合统计。
- **时间维度分析**：解锁 `groupBy: "month"`，支持使用 PostgreSQL 的 `to_char` 按月统计趋势，绘制折线图。
- ** BigInt 安全处理**：捕获并转换 PG 驱动聚合运算返回的 `BigInt`，防止 SSE JSON 序列化崩溃。

### 2.5 前端渲染引擎与 UX 深度打磨
解决 Agent 架构落地时的典型 UI 痛点：
- **Markdown 强化**：引入 `remark-gfm` 插件，确保 LLM 输出的 GFM 表格 100% 稳定渲染。配合 Tailwind Typography 插件，实现精美的排版。
- **动态数据表格**：废弃写死表头的组件，改为完全数据驱动的 `CollapsibleTable`。自动提取 JSON Keys 作为表头，智能判断数字类型右对齐，并支持长表格折叠/展开。
- **状态归一去重**：前端对连续相同的 `tool_call` 进行合并显示（如“已完成：query_sales (共 3 次)”），并通过 Prompt 约束禁止 LLM 重复画表，保持界面清爽。

## 3. 踩过的坑与经验教训

| # | 问题 | 根因与解决 |
|---|------|-----------|
| 1 | **Prisma 生成报错 EPERM** | Windows 系统下 Node 进程死锁 `.dll` 文件。解决：`taskkill /F /IM node.exe` 后重新生成。 |
| 2 | **SSE 流中断：Do not know how to serialize a BigInt** | PostgreSQL 驱动将 `SUM()` 结果作为 `BigInt` 返回，原生 `JSON.stringify` 不支持。解决：在 Kysely 查询结果返回前，显式调用 `Number()` 转换。 |
| 3 | **AI 回复“因工具限制无法获取每日明细”** | 工具的 Zod Schema 未开放时间维度。解决：在 Schema 中增加 `month` 枚举，并在 Kysely 中实现按月聚合。 |
| 4 | **Markdown 表格渲染不稳定** | `react-markdown` 默认不支持 GFM 表格。解决：安装并配置 `remark-gfm` 插件。 |
| 5 | **界面出现多个重复表格和状态** | LLM 多次调工具且自作聪明画表。解决：前端合并相同 Tool 状态，Prompt 红线禁止 LLM 画已有数据表。 |

## 4. 架构现状与下一步规划

经过 Phase 11 的改造，系统已具备**标准业务模型底座**与**复杂关联分析能力**。
底层的 Kysely 封装使得数据库操作不再僵硬，这为我们接下来的目标打下了坚实基础：

- **Phase 12: 会话持久化与多轮记忆**：将对话历史和工具调用记录存入数据库，支持左侧多会话列表切换。
- **Phase 14: 动态数据源与半自动映射 (终极 SaaS)**：实现在前端配置外部数据库连接，抓取外部 Schema，并通过配置界面将标准概念映射到真实字段，最终通过 Kysely 动态执行外部查询。
